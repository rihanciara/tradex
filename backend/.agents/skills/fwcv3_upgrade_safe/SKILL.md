# FWCV3 Upgrade-Safe Module Development

When developing features for FWCV3 modules that alter core system behaviors (such as Ledger math, transaction totals, or core computations usually found in `App\Utils` like `TransactionUtil`), **NEVER** copy the entire parent method into the child module to modify it inline.

Because FWCV3 is constantly updated by the base vendor (currently v12+), simply copy-pasting a 300+ line function means your module will immediately fall out of sync when the core vendor adds new structural variables (e.g., `$total_reverse_payment`, new parameters like `$hms_booking`), causing catastrophic "Undefined Variable" or "Undefined Array Key" errors.

## The Solution: The Proxy Pattern

Instead of overriding the entire method, use the **Proxy Pattern** inside your Module's Service injected class (e.g. `JerryTransactionUtil`).

### Step 1: Call the Parent
Run the core FWCV3 logic first to harvest all the natively upgraded, complex array builders.

```php
public function getLedgerDetails($contact_id, $start, $end, $format = 'format_1', $location_id = null, $line_details = false)
{
    // Check if the module feature is toggled off
    if (\App\System::getProperty('jerry_ledger_fix') != '1') {
        return parent::getLedgerDetails($contact_id, $start, $end, $format, $location_id, $line_details);
    }

    // 1. Call the PARENT strictly to get the 100% up-to-date base logic
    $output = parent::getLedgerDetails($contact_id, $start, $end, $format, $location_id, $line_details);

    // ... continue to Step 2
```

### Step 2: Extract Module-Specific Data Natively
Run whatever isolated, custom queries your module needs (such as fixing Advance Payments).

```php
    // 2. Fetch missing or customized variables targeting ONLY your specific feature
    $overall_total_advance_payment = $this->__paymentQuery(...)
        ->where('is_advance', 1)
        ->sum('amount');
```

### Step 3: Amend the Output Array
Overlay your mathematical corrections directly onto the result of the parent array before passing it back to the view.

```php
    // 3. Apply custom math adjustments over the top
    $output['balance_due'] = $output['total_invoice'] + $output['total_purchase'] - $output['total_paid'] + $output['beginning_balance'];
    
    $output['all_invoice_paid'] += $overall_total_advance_payment;
    $output['all_balance_due'] -= $overall_total_advance_payment;

    return $output;
}
```

### Why this works:
- Any new features added by FWCV3 (like gym subscriptions, HMS bookings, specific new array keys like `total_reverse_payment`) are automatically generated by `parent::getLedgerDetails()` and blindly passed into your `$output`.
- Your module only alters the exact mathematical fields that your business logic specifies.
- It survives all future FWCV3 version upgrades with ZERO maintenance required.

## Global Settings & FOUC

Native modules in FWCV3 can append JavaScript or CSS to the `<head>` using `DataController::get_additional_script()`.

- **To fix White Screen Flashes (FOUC)**: Do **NOT** use the `'additional_html'` key in the return array of `get_additional_script()`. That injects directly before the `</body>` tag, causing the screen to render white before snapping to styled CSS.
- **Instead**, dump your `<style>` block into the `'additional_css'` key. This injects it directly into the `<head>` of `layouts.app.blade.php`, rendering it synchronously and ending the FOUC.

```php
public function get_additional_script()
{
    $css = '<style> html { filter: invert(...); } </style>';
    return [
       'additional_html' => '',
       'additional_css' => $css,
    ];
}
```

## Frontend UI Injection (The Hook Pattern)

When you need to add new buttons, modals, or offline logic to the core V12 views (like the POS screen `resources/views/sale_pos/create.blade.php`), **NEVER** edit the core `.blade.php` or `pos.js` files directly.

### Step 1: Use Module Hooks
Ultimate POS triggers view hooks via `$this->moduleUtil->getModuleData('pos_settings')` or similar view arrays (e.g., `pos_after_table`). Use your module's `DataController.php` or `Http/Controllers` to return a rendered view from your module.

```php
// In JerryUpdates/Http/Controllers/DataController.php
public function modify_pos_tool_box() {
    return view('jerryupdates::javascript_tweaks');
}
```

### Step 2: Inject HTML via jQuery
Inside your module's Blade file (e.g., `javascript_tweaks.blade.php`), use a `setInterval` or `setTimeout` to wait for the core `pos.js` to finish loading, and then use jQuery to safely append or prepend your custom HTML to existing DOM elements.

```javascript
// In JerryUpdates/Resources/views/javascript_tweaks.blade.php
$(document).ready(function() {
    // Wait for V12 pos_form_actions to render
    setTimeout(function() {
        if ($('#pos_table').length > 0) { // Ensure we are on the POS screen
            
            // Inject 'All Sales' button upgrading safe
            if ($('#jerry-all-sales-btn').length === 0) {
                var allSalesHtml = '<a href="/sells" id="jerry-all-sales-btn" class="tw-font-bold tw-text-gray-700 tw-flex tw-flex-col tw-items-center"> <i class="fa fa-backward"></i> All Sales </a>';
                
                // Prepend to the action container safely
                $('.pos-form-actions .tw-overflow-x-auto').prepend(allSalesHtml);
            }
        }
    }, 1000);
});
```

### Step 3: Network Interception ($.ajaxPrefilter)
If you need to fundamentally change how Native V12 handles form submissions (like saving POS sales offline), do not override `$scope.pos_save()`. Instead, intercept the outgoing global network traffic.

```javascript
$.ajaxPrefilter(function(options, originalOptions, jqXHR) {
    if (!options.type || options.type.toUpperCase() !== 'POST') return;
    if (options.url.indexOf('/pos') === -1) return;

    var originalError = options.error;
    
    options.error = function(xhr, status, error) {
        if (status === 'timeout' || xhr.status === 0 || !navigator.onLine) {
            // Catch the failure BEFORE V12 crashes
            console.log("Offline intercepted. Saving locally...");
            
            // Force reset the V12 UI to native state
            if (typeof enable_pos_form_actions === 'function') {
                enable_pos_form_actions();
            }
            return false; // Prevent V12 from showing error toast
        }
        if (originalError) originalError.apply(this, arguments);
    };
});
```

## Frontend Blade Proxy (Dual-Mode Architecture)

When overriding core Blade templates via the `custom_views/` directory, you risk breaking the UI if the vendor updates the core files. To mitigate this while keeping your old custom code, use a **Dual-Mode Proxy**.

### The Hybrid Wrapper
Wrap your entire custom Blade file in an `@if` block that checks for a `jerry_traditional_mode` setting. If disabled, it should automatically fall back to the native vendor file using `extract(get_defined_vars())`.

```blade
@if(\Modules\JerryUpdates\Utils\JerrySettings::get('jerry_traditional_mode', '1') == '1')
    {{-- YOUR TRADITIONAL CUSTOM HTML CODE HERE --}}
    @extends('layouts.app')
    ...
@else
    {{-- UPGRADE-SAFE FALLBACK --}}
    @php
        extract(get_defined_vars());
        include resource_path('views/your/path/to/file.blade.php');
    @endphp
@endif
```

### Why this is essential:
1. **Zero-Downtime Rollbacks**: If a vendor update breaks your custom view, the user can instantly toggle "Traditional Mode" OFF to restore the clean native UI.
2. **Side-by-Side Development**: You can maintain your heavy custom logic (Traditional) while slowly building out JS-based injections (Upgrade-Safe) on top of the native file in the background.
3. **No Merge Conflicts**: Since `custom_views` are checked first by the framework, this proxy acts as a "smart gatekeeper" that decides whether to run your code or the vendor's code at runtime.

By combining the **Backend PHP Proxy**, **Frontend JS Hooks**, and **Blade Proxy Wrappers**, you achieve a completely decoupled, upgrade-proof architecture.
